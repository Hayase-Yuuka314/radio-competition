# 通信协议与编码方式对系统架构的影响分析

> 本文分析项目中喷泉码、DSSS 扩频、Gold 码 CDMA、TDD 时分等关键设计选择如何影响整体架构。

---

## 一、系统分层架构

```
┌─────────────────────────────────────────────────────────────┐
│  应用层：文件传输（裁判提供的随机二进制文件 → 正确恢复）      │
├─────────────────────────────────────────────────────────────┤
│  传输层：Fountain Code (LT 码, Robust Soliton 度分布)        │
│          ● 无 ACK 确认                                      │
│          ● 速率自适应（收够即停，不需协商码率）                │
│          ● 对丢包率不敏感（30% 丢包只多要 ~5% 修复包）        │
├─────────────────────────────────────────────────────────────┤
│  MAC 层：TDD + CCA + 跳频                                   │
│          ● 时分双工（单滤波器约束：TX 时 RX 关闭）             │
│          ● 空闲信道评估（CSMA/CA 指数退避）                   │
│          ● 16 通道跳频 + 信道质量评分 + 黑名单                │
├─────────────────────────────────────────────────────────────┤
│  物理层：DSSS 扩频 (SF=128) + Gold 码 (CDMA 多址)            │
│          ● 处理增益 21 dB（抗干扰/抗噪声）                    │
│          ● Gold 码三值互相关（多队同时同频互不干扰）           │
│          ● 前导 + 同步字 + 两阶段帧解码                       │
├─────────────────────────────────────────────────────────────┤
│  硬件层：PlutoSDR (AD9363) + 2.4GHz/433MHz 带通滤波器        │
│          ● 两台 SDR：一台接滤波器发射，一台裸接收              │
│          ● TX 与 RX 永不同时（TDD 保证）                      │
└─────────────────────────────────────────────────────────────┘
```

## 二、喷泉码（Fountain Code）对架构的影响

### 2.1 为什么选喷泉码而非 ARQ

| | ARQ (自动重传) | Fountain Code (喷泉码) |
|---|---|---|
| **确认机制** | 每包需要 ACK/NAK | **无需任何 ACK** |
| **丢包影响** | 丢包 → 重传 → 延迟翻倍 | 丢包 = 少收一个，多收几个即可 |
| **丢包率 30% 时** | 有效吞吐 = 原始 × 0.7 × 0.7 = **49%**（含 ACK 也丢） | 只需多收 ~5% 包 → 效率 **~95%** |
| **双向信道要求** | 必须有回传信道 | **单向即可**（TX 只管发，RX 只管收） |
| **码率协商** | 需要 CSI 反馈自适应 | **无速率**（rateless）→ 发到对方收够为止 |
| **对抗随机干扰** | 差（ACK 本身被干扰 → 超时） | **天然鲁棒** |

### 2.2 喷泉码如何改变每一层

**MAC 层**：
- 无 ACK 意味着 **TX 可以全力发射**，不需要预留 RX 窗口收确认
- TDD 中的 RX 时隙仅用于接收本方数据（喷泉包直达），不是收 ACK
- CCA 退避后无需等待 ACK 超时，退避时间可以更短
- `CCAMode.OFF` 激进模式（无 CCA）可行——因为各队互不干扰（CDMA），无须检测信道

**传输层代价**：
- 引入 5~15% 修复包开销（收 K+epsilon 包才能解码）
- 解码需要 Peeling + Gaussian Elimination 两级，CPU 开销略高于 ARQ 的 CRC 校验
- 但在我们的测试中开销接近 0%（系统包全收，不依赖修复包）

**物理层**：
- 喷泉包 = 20 字节头 + `block_size` 字节负载
- 包大小固定 → DSSS 帧长度可预测 → 帧检测更可靠
- 包 ID 通过 `_derive_neighbors(packet_id, k, file_id)` 确定性生成度分布 → 无需传输邻居表

**系统码优化**：
- 发前 K 个包时直接发原始块（systematic phase）
- 如信道完美 → 零开销解码（收 K 个系统包直接拼接）
- 如有丢包 → 修复包补足

## 三、DSSS 扩频 (SF=128) 对架构的影响

### 3.1 处理增益原理

```
处理增益 (dB) = 10 × log₁₀(扩频因子)

SF=128:  PG = 10 × log₁₀(128) ≈ 21.0 dB

含义：解扩后信噪比 = 接收端信噪比 + 21 dB
```

**实际效果**：接收端 SNR = 5dB 时，解扩后有效 SNR = 5 + 21 = 26dB → 几乎无误码

### 3.2 架构代价

| 代价 | 量化 | 影响 |
|---|---|---|
| **带宽效率** | 1/128 = 0.78% | 2MHz 带宽只传 ~15.6 kbps 原始比特 |
| **处理复杂度** | 每比特 128 次乘加 | CPU 密集但向量化 (np.dot) 可接受 |
| **同步要求** | 码片级对齐 | 需要前导 + 同步字两阶段检测 |
| **频谱占用** | 全带宽扩频 | 每队占满 2MHz → 16 队共享时需 CDMA 隔离 |

### 3.3 为什么 SF=128 够了

```
Scaling:
  SF=32   → PG=15dB  → SNR=5dB 时有效 SNR=20dB（边缘）
  SF=64   → PG=18dB  → SNR=5dB 时有效 SNR=23dB（OK）
  SF=128  → PG=21dB  → SNR=5dB 时有效 SNR=26dB（充裕）
  SF=256  → PG=24dB  → 吞吐减半（不值得）
```

我们的测试在 SNR=5dB 全部通过 → SF=128 是吞吐与鲁棒性的最佳平衡点。

## 四、Gold 码 CDMA 对架构的影响

### 4.1 多址原理

```
每个 team_id 对应一个 Gold 码循环移位：

team 0: Gold(mseq1 × mseq2_shift[0])   → shift = 0 × 128 = 0
team 1: Gold(mseq1 × mseq2_shift[128]) → shift = 128
team 2: Gold(mseq1 × mseq2_shift[256]) → shift = 256
...

互相关上界: |CC| ≤ 65 / 1023 ≈ 0.0635 (n=10)
```

**这意味着**：8 队同时同频发射时，任意两队之间的码互相关 < 6.35%，其余队的信号对你来说是"宽带低功率噪声"。

### 4.2 对架构的影响

| 层面 | 影响 |
|---|---|
| **物理层** | 每队需要独立的 Gold 码生成器（`team_id → shift` 映射） |
| **MAC 层** | 不需要 TDMA/FDMA 分割资源，多队天然同频共存 |
| **信道** | 多队信号叠加 = 信干比 (SIR) ≈ 1/(N-1) 在白噪声基础上增加干扰 |
| **接收端** | 每个 RX 只需自己队伍的码解扩 → **自动过滤其他队伍信号** |
| **扩展性** | 理论上 16 队同频 (SF=128 空间有 1023 个不同码) |

### 4.3 仿真验证

```
8 队同频 SNR=15dB:  [########]  8/8 全部通过
8 队同频 SNR=20dB:  [########]  8/8 全部通过
```

**原因**：每队解扩后，其他 7 队信号被 21dB 处理增益 + Gold 码隔离双重压制，等效 SIR ~ 21 + 17 = 38dB → 几乎无影响。

## 五、TDD MAC 对架构的影响

### 5.1 为什么必须用 TDD

```
硬件约束：只有 1 个 2.4GHz 滤波器

方案 A：滤波器给 RX → TX 杂散污染所有队伍 ❌
方案 B：滤波器给 TX → RX 需要保护 ✓

TDD 保护机制：
  TX 时隙：TX SDR 发射，RX SDR 增益降至 -20dB（保护前端）
  RX 时隙：TX SDR 静默，RX SDR 正常接收
```

### 5.2 时隙分配

```
TX 70ms ─── Guard 5ms ─── RX 40ms ─── Guard 5ms = 120ms 超帧

占空比：TX 58% / RX 33% = 1.75:1
    → TX 时间接近 RX 的 2 倍（喷泉码无 ACK，TX 应更多）
```

### 5.3 CCA 的双重作用

| 作用 | 说明 |
|---|---|
| **防止碰撞** | 检测到信道忙 → 指数退避 → 换时间发射 |
| **信道选择** | 连续 5 次 CCA 失败 → 黑名单该信道 → 跳频 |

CCA 在喷泉码场景下可关闭 (`CCAMode.OFF`)——各队同时发射是预期的，CDMA 已提供隔离。

## 六、协议栈各层交互与边界

### 6.1 数据流动

```
原始文件 (N bytes)
  │
  ▼  FountainEncoder(block_size=128)
  │   → K = ceil(N/128) 个块
  │   → 每个喷泉包 = [20B 头 | 0~128B 负载]
  │   → 无限流（系统包 → 修复包）
  │
  ▼  ContestDSSSEncoder(team_id, SF=128)
  │   → 每包：前导(256chips) + 同步字(128chips) + 负载(包字节×8×128chips)
  │   → IQ 采样输出
  │
  ▼  TDDController
  │   → CCA 检测 → TX 时隙 → 发射 IQ 到 SDR
  │
  ▼  PlutoSDR TX (+滤波器) ~~~ 空中 ~~~
  │
  ▼  PlutoSDR RX (无滤波器)
  │
  ▼  TDDController
  │   → RX 时隙 → 接收 IQ
  │
  ▼  ContestDSSSDecoder
  │   → 滑动相关找前导 → 同步字验证 → 解扩 → 提取喷泉包
  │
  ▼  FountainDecoder
  │   → Peel: 度=1 包剥皮恢复块
  │   → Gaussian Elim: 卡住时解方程组
  │   → 块拼接 → 恢复文件
  │
  ▼  恢复文件 (N bytes, SHA256 一致)
```

### 6.2 层间接口约束

| 接口 | 发送端 | 接收端 | 约束 |
|---|---|---|---|
| 应用→传输 | 文件字节 | 恢复字节 | `block_size` 必须一致 |
| 传输→物理 | 喷泉包 (20+N bytes) | 喷泉包 | 包头含 `block_size` 字段 |
| 物理→MAC | IQ 采样 (complex64) | IQ 采样 | `sample_rate`、`spreading_factor` 必须一致 |
| MAC→硬件 | IQ 采样 | IQ 采样 | 频点、增益由 MAC 控制 |
| 物理层内 | Gold 码 | Gold 码 | `team_id` 唯一确定码 |

## 七、传统 FEC (卷积码) vs 喷泉码

### 7.1 分工

| | 卷积码 (ADR-002) | 喷泉码 (ADR-005) |
|---|---|---|
| **用途** | 仿真链路（传统 BPSK 模式） | **比赛链路** |
| **位置** | 物理层内（比特级前向纠错） | 传输层（包级纠删） |
| **纠错粒度** | 比特 | 包/块 |
| **是否需要 ACK** | 是（配合 ARQ） | **否** |
| **信道适应性** | 需预先设定码率 | 自适应（rateless） |

**二者互补而非替代**：卷积码纠正比特错误，喷泉码恢复丢失的包。比赛中主要依赖喷泉码——比特级错误由 DSSS 处理增益 + Gold 码隔离压制到可忽略水平。

## 八、性能边界与极限

### 8.1 单链路极限

| 参数 | 值 | 计算公式 |
|---|---|---|
| 原始比特率 | 15.6 kbps | 2 MHz ÷ 128 (SF) |
| 有效吞吐（喷泉 0% 开销） | 15.6 kbps × 128/148 ≈ **13.5 kbps** | 包效率 = payload/(header+payload) = 128/148 |
| TX 占空比 (TDD) | 58% | 70ms ÷ 120ms |
| **最终有效吞吐** | 13.5 × 0.58 ≈ **7.8 kbps** | 含 TDD 时隙损失 |
| 最低可通 SNR | **5 dB** | 实测 |

### 8.2 多队扩展性

| 队数 | 每队 SIR | Gold 隔离余量 | 实测 |
|---|---|---|---|
| 2 | 0 dB | 21+17=38 dB 远大于 0 dB | ✅ |
| 4 | -4.8 dB | 38 dB 远大于 4.8 dB | ✅ |
| 8 | -8.5 dB | 38 dB 远大于 8.5 dB | ✅ |
| 16 | -11.8 dB | 38 dB >> 11.8 dB | 理论可达 |

**SIR 最差情形（16队）**：其他 15 队每队信号强度相同 → 叠加后干扰功率 = 15 × 信号功率 → SIR = -11.8dB。Gold 码处理增益 21dB + 互相关隔离 ~17dB = 38dB 压制 → 等效 SIR = -11.8+38 = **26.2dB** → 绰绰有余。

## 九、总结：关键设计选择与影响矩阵

| 选择 | 解决了什么问题 | 付出的代价 | 对架构的最大影响 |
|---|---|---|---|
| **喷泉码** | 无需 ACK，抗拒高丢包 | 5~15% 修复包开销 | MAC 不需要 ACK 窗口，TX 可激进发射 |
| **DSSS SF=128** | 21dB 抗干扰增益 | 带宽效率 1/128 | 物理层计算密集，需码片同步 |
| **Gold 码 CDMA** | 多队同频共存不互扰 | 码空间受限（1023 个码） | 接收端自动过滤其他队信号 |
| **TDD 单滤波器** | 硬件约束下保护 RX | TX 占空比 ~58% | TX/RX 严格分时，CCA 可选关闭 |
| **标准 Gold 码** | 保证三值互相关 | 需正确实现 LFSR | 码生成统一到 `gold_code.py` |
