# DECISIONS — 架构决策记录

## ADR-001：选择单载波 BPSK 作为初始基线

- **状态**: accepted
- **背景**: 需要快速建立可测量基线，确保后续 ML 增强有可靠的 fallback。
- **候选方案**: 单载波 BPSK、单载波 QPSK、OFDM。
- **评估场景**: 理想信道、AWGN、CFO。
- **结果**: BPSK 实现最快、同步最简单、调试最直观。
- **决策**: 第一版使用单载波 BPSK，稳定后升级为 QPSK，OFDM 作为后期可选分支。
- **代价**: 峰值吞吐率较低，但不影响先建立稳定基线。
- **回退条件**: 不适用（这是起点）。

## ADR-002：使用卷积码 (rate 1/2, K=7) 作为首版 FEC

- **状态**: accepted
- **背景**: 需要一种比重复码更有效的 FEC，同时实现复杂度可控。
- **候选方案**: 无编码、重复码、卷积码、LDPC。
- **评估场景**: AWGN 信道。
- **结果**: 卷积码实现相对直接，维特比译码器 CPU 友好。
- **决策**: 首版使用 NASA-DSN 标准卷积码。
- **代价**: LDPC 可能提供更好性能，但开发成本高，作为后期比较。
- **回退条件**: 若实时性不足，降为更高码率或减少迭代。

## ADR-003：Python 3.11 + NumPy/SciPy 作为核心工具链

- **状态**: accepted
- **背景**: 需要在快速开发和运行效率间取得平衡。
- **候选方案**: Python、C++/GNU Radio、MATLAB。
- **决策**: Python 3.11 + NumPy 向量化可满足实时处理需求，CPU 优先不考虑 GPU。
- **代价**: 纯 Python 某些 DSP 操作较慢，关键路径可能需要 C 扩展或 Numba。
- **回退条件**: 实测实时因子 > 1 时考虑 C 扩展。

## ADR-004：ML 模型必须有无模型回退路径

- **状态**: accepted
- **背景**: 计划文档明确要求 ML 失败时自动回退。
- **决策**: 所有 ML 输出都必须经过置信度门控、OOD 检测和滞回，失败时回到 P1 Robust Profile。
- **代价**: 增加了系统复杂度，但保证了比赛可靠性。
- **回退条件**: 不适用（这是约束而非选择）。

## ADR-005：用喷泉码 (LT/Raptor) 替代 ARQ 确认重传

- **状态**: accepted
- **背景**: 比赛场景中，多队同时同频竞争，信道极度不可靠。传统 ARQ 需要双向确认，ACK 本身可能丢失，导致效率低且链路复杂。
- **候选方案**: ARQ (逐包确认)、喷泉码、混合 ARQ。
- **评估**: ARQ 在丢包率 >30% 时吞吐量急剧下降。喷泉码 (LT code with Robust Soliton distribution) 对丢包率不敏感，只需收够 K+~5% 包即可解码，天然适合无 ACK 的单向广播信道。
- **决策**: 使用 LT 码作为喷泉码实现。系统码阶段先发 K 个原始块，再发修复包。解码器 Peel + Gaussian elimination 两级。
- **代价**: 引入了约 5-15% 的额外传输开销（修复包），但消除了双向协议复杂度。
- **回退条件**: 若喷泉码解码成功率 < 90% 在目标 SNR 下实测，回退到 ARQ + FEC。

## ADR-006：统一为标准 Gold 码生成 (优选对多项式)

- **状态**: accepted
- **背景**: 原代码库存在两套 Gold 码生成器：
  - `dsss.py`: 简化版，用 `np.random.default_rng(seed)` 随机生成 LFSR 初值 → 无互相关保证。
  - `gold_code.py`: 标准版，使用优选对多项式 (Dixon, 1994) + 循环移位 → 三值互相关 {-1, -t(n), t(n)-2}，但 `_lfsr_step` 抽头索引 bug (`t-1` 应为 `n-t`) 导致实际只产生 21/63 短周期。
- **修复**:
  1. 修正 `_lfsr_step` 抽头索引为 `n - t`。验证 n=10 产生完整 1023 周期序列。
  2. 为 n=5..10 添加完整优选对多项式表，`generate_standard_gold_code` 自动选择最小满足的 n。
  3. `dsss.py` 的 `generate_gold_code(seed)` 和 `generate_spreading_code(team_id)` 改为委托 `gold_code.py`。
  4. `contest/dsss_pipeline.py` 统一从 `dsss.py` 导入，消除双重依赖。
- **决策**: 全项目统一为 `gold_code.py` 的标准 Gold 码。`dsss.py` 作为高层 API 入口，内部委托到标准实现。
- **验证**: 21 个 adversarial 测试全部通过；平衡性测试修正为允许三值相关的合法偏差 (max 65/1023 ≈ 6.35%)。
- **回退条件**: 不适用（这是 bug 修复 + 架构统一，旧代码有缺陷）。

## ADR-007：TDD + 滤波器专用于发射链路

- **状态**: accepted
- **背景**: 比赛仅提供 1 个 2.4GHz 带通滤波器。若滤波器给接收机，发射机杂散会影响其他队伍及自机的接收机（电源/地线耦合）。若滤波器给发射机，接收机无外部滤波但可通过 TDD 保护。
- **决策**: 2.4GHz 滤波器专用于发射链路。接收链路靠 TDD 时间隔离 + 增益回退 + 正交极化天线保护。TDD superframe: CCA (5ms) → Guard (5ms) → TX (80ms) → Guard (5ms) → RX (80ms) → Guard (5ms)。
- **代价**: 接收灵敏度略低于带滤波器方案，但可避免发射杂散污染。
- **回退条件**: 现场若 2.4GHz WiFi 干扰严重，切换至 433MHz 备用频段。
