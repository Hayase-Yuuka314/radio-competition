> 版本：v1.0

> 状态：规划基线，等待完整赛题规则与硬件实测参数补全

## 0. 给执行 Agent 的最高优先级指令

本节优先级高于本文档其余技术建议。Agent 在开始编码前必须完整阅读本节。

### 0.1 最终任务

构建一套能够在共享频谱、有限资源、复杂电磁干扰条件下，通过两台兼容 PlutoSDR 的 NanoSDR 设备完成随机二进制文件无线传输的系统，并尽可能提高规定比赛时间内正确恢复的有效数据量。

系统应由以下部分组成：

1. 可重复的纯软件通信仿真器；
2. 可靠的传统数字通信收发基线；
3. 可配置的干扰与信道模型；
4. 完整的数据集生成、记录和划分流程；
5. 机器学习干扰识别器；
6. 有置信度门控和保底策略的自适应接收机；
7. NanoSDR/PlutoSDR硬件接入层；
8. 自动测试、压力测试、对照实验和结果报告；
9. 可选的自适应发射或强化学习模块，但它们不得阻塞稳定基线交付。

### 0.2 Agent 必须遵守的工作方式

1. **先建立可测量基线，再加入机器学习。**不得从端到端神经网络直接解调文件开始。
2. **每次只引入一个主要变量。**例如增加纠错编码时，不得同时更换调制、同步和均衡算法。
3. **每个模块都必须有输入、输出、单位、数据类型、异常行为和测试。**
4. **任何“更好”的结论必须由统一测试矩阵支持。**不得仅凭单次演示、训练准确率或主观波形判断。
5. **最终评价指标是 Goodput，而不是模型 Accuracy。**机器学习准确率只能作为诊断指标。
6. **对未知赛制参数使用显式 `null/TODO`。**不得自行编造频率、带宽、功率、比赛时长、反馈权限或评分细则。
7. 规则未确认时，可以继续进行纯软件仿真和单元测试，但不得把假定参数写死在算法中。
8. **所有随机过程必须可复现。**保存随机种子、配置、代码版本、模型版本和数据版本。
9. **机器学习必须有无模型回退路径。**模型加载失败、推理超时、置信度不足或输入异常时，自动回到稳定接收配置。
10. 优先实现 CPU 可实时运行的小模型；不假设比赛电脑有可用 NVIDIA GPU。
11. 不因追求架构完整而忽略实时数据流、缓冲区溢出、丢样、驱动断连和进程崩溃。
12. 不执行越权外部操作：未经用户明确授权，不推送远程仓库、不部署到外部系统、不进行开放空间射频发射。
13. 实机联调时必须遵守组委会规定和当地无线电规范。优先使用仿真、屏蔽环境或“同轴线缆 + 合适衰减器”；不得把发射端直接无衰减连接到接收端。
14. 不得使用比赛禁止的外部通信接口、网络服务或云端推理。最终系统必须可完全离线运行。
15. 每完成一个阶段，都必须更新：
    - `STATUS.md`：当前进展、通过的测试、当前阻塞；
    - `DECISIONS.md`：关键设计选择、证据和被否决方案；
    - `experiments/registry.csv`：实验配置、代码版本、数据版本和结果；
    - 本文档中的里程碑状态或项目看板。

### 0.3 Agent 的停止与询问条件

出现以下情况时，Agent 应停止相应高风险动作并向用户询问，但仍可继续不依赖该信息的离线工作：

- 完整规则与截图表述冲突；
- 不清楚是否允许反向反馈或收发角色切换；
- 不清楚允许的频点、带宽、发射增益或输出功率；
- 不清楚是否允许携带预训练权重和第三方库；
- 需要进行未经许可的空口发射；
- 计划中的硬件 API 与实际 NanoSDR 不兼容；
- 需要删除用户数据、覆盖未知文件或执行其他不可恢复操作；
- 两个候选技术路线会显著改变工期或硬件要求，且现有证据无法选择。

### 0.4 完成声明约束

Agent 只有在相应验收测试通过后才能声称某阶段“完成”。以下情况不得视为完成：

- 代码能运行但没有测试；
- 只在单一随机种子下成功；
- 只在理想 AWGN 信道下成功；
- 只输出训练集准确率；
- SDR 能收到波形但无法逐字节恢复文件；
- 平均值提升但失败率或最坏情况显著恶化；
- 模型有效但推理速度低于实时数据到达速度；
- 手工调整一次参数后成功，但无法自动复现。

## 1. 执行摘要

根据现有赛题截图，比赛模拟共享频谱、有限资源和复杂电磁干扰下的卫星通信场景。每支队伍使用一台 SDR 发射、另一台 SDR 接收组委会提供的随机数据文件；多支队伍同时工作并互相形成有源干扰；裁判按正确解调的随机数据量排名。

因此，工程目标不是追求某个孤立算法的最高准确率，而是最大化：

\[
G=\frac{N_{\text{correct payload bits}}}{T_{\text{contest}}}
\]

其中 \(G\) 是有效吞吐率 Goodput，分子仅计算被正确恢复并通过完整性校验的有效载荷，不包括前导码、导频、包头、纠错冗余、重复包和错误数据。

推荐采用“混合式智能通信系统”：

- 底层使用成熟、确定性的数字通信算法完成组帧、同步、滤波、均衡、解调、纠错和完整性校验；
- 上层机器学习识别干扰类型、预测接收策略或信道质量；
- 策略控制器只在允许范围内调整接收参数；
- 如果规则允许可靠反馈，再扩展到调制编码方案、包长、频点和发射策略的闭环调整；
- 强化学习仅作为后期可选模块，不得取代稳定基线，也不得在正式比赛中进行高风险随机探索。

建议的首个可交付智能功能是：

> **手工频谱/相关特征 + 随机森林干扰分类 + 置信度门控 + 自适应陷波/带通/同步阈值选择。**

该方案计算量低、可解释、CPU 可实时运行、对数据量要求相对较低，并能与传统接收机自然结合。

## 2. 赛题信息、假设与待确认事项

### 2.1 从现有截图中能够确认的事实

以下条目视为当前已知事实，但在获得完整竞赛规则后仍应复核：

1. 赛项名称为“电波对决争锋”。
2. 场景强调共享频谱、有限资源和复杂电磁干扰。
3. 重点考查通信系统设计、协议优化、链路维护、抗干扰传输和有效数据交互。
4. 每支队伍必须使用组委会统一提供的设备，自主设计并搭建通信系统。
5. 在规定频段内使用一台 SDR 发射、另一台 SDR 接收。

7. 多支队伍同时同场收发，各队发射信号会对其他队伍形成有源干扰。
8. 参赛电脑除 SDR 外的所有通信接口会被关闭。
9. 排名依据是正确解调的随机数据量，正确数据量越高排名越高。
10. 组委会在培训前向每队提供：

    - 2.4 GHz 带通滤波器 ×1；

## 3. 项目目标、指标与优化边界

### 3.1 一级目标

最大化规定时间内正确恢复的有效载荷比特数：

\[
N_{\text{correct}}=\sum_{i=1}^{K} L_i\cdot\mathbb{1}(\text{block}_i\text{ passes integrity check})
\]

其中 \(L_i\) 是第 \(i\) 个有效载荷块长度，\(\mathbb{1}\) 为正确性指示函数。

### 3.2 二级工程指标

每次实验至少记录：

| 指标 | 符号/单位 | 用途 |
|---|---:|---|
| 有效吞吐率 | bit/s 或 byte/s | 最终主指标 |
| 原始比特错误率 | pre-FEC BER | 衡量解调质量 |
| 译码后比特错误率 | post-FEC BER | 衡量纠错效果 |
| 包错误率 | PER/FER | 衡量完整数据块成功率 |
| 同步捕获概率 | % | 衡量前导码与同步鲁棒性 |
| 平均捕获时间 | ms | 衡量启动和重同步速度 |
| 误检率/漏检率 | % | 衡量帧检测器 |
| 丢样数量 | samples | 判断是否满足实时性 |
| 实时因子 | processing time / signal duration | 必须小于 1，并保留裕量 |
| CPU/GPU 占用 | % | 现场可运行性 |
| 峰值内存 | MB/GB | 防止崩溃 |
| 端到端时延 | ms | 若规则实时计分则重要 |
| 模型置信度 | 0–1 | 策略门控 |
| 策略切换次数 | 次/min | 防止抖动 |
| 第 5 百分位 Goodput | bit/s | 衡量弱场景稳定性 |
| 完全失败率 | % | 衡量灾难性风险 |

### 3.3 多目标决策顺序

设计冲突时按照以下顺序权衡：

1. 遵守规则与射频安全；
2. 保证能够恢复正确数据；
3. 降低完全失锁和进程崩溃概率；
4. 提高低分位数和最坏情况下的 Goodput；
5. 提高平均 Goodput；
6. 降低时延和资源占用；
7. 提升模型准确率或算法新颖性。

### 3.4 明确的非目标

- 不以发表论文或使用最复杂模型为首要目标；
- 不从零训练大型神经网络；
- 不把随机文件的内容规律作为预测目标；
- 不依赖互联网、云端 API 或远程服务器；
- 不依赖正式赛场中在线重新训练深度模型；
- 不在传统基线尚未稳定时开发端到端神经接收机；
- 不为了平均吞吐率牺牲不可接受的失败率；
- 不在规则未明确时实现可能违规的主动干扰或超功率策略。

## 4. 总体技术路线

### 4.1 推荐架构

```mermaid
flowchart TD
    A["随机二进制文件"] --> B["组帧、CRC、FEC、交织"]
    B --> C["调制、脉冲成形、SDR发射"]
    C --> D["共享信道、多队干扰、硬件损伤"]
    D --> E["SDR采样、同步、滤波、均衡"]
    E --> F["解调、译码、去重、文件恢复"]
    E --> G["特征提取与干扰分类"]
    G --> H["置信度门控与策略选择"]
    H --> E
```

### 4.2 分层职责

| 层级 | 主要职责 | 首选方法 |
|---|---|---|
| 文件层 | 分块、序号、去重、重组、完整性 | 确定性协议 |
| 链路层 | 帧结构、CRC、FEC、交织、可选重传 | 传统通信协议 |
| 物理层 TX | 调制、成形、导频、发射 | BPSK/QPSK 起步 |
| 信道层 | AWGN、多径、频偏、干扰、硬件损伤 | 可配置仿真 + 实测 |
| 物理层 RX | AGC、同步、滤波、均衡、解调、译码 | 传统 DSP |
| 感知层 | 频谱、相关、质量和历史统计 | 手工特征/小型 CNN |
| 决策层 | 选择接收配置和可选链路模式 | 规则表/监督学习 |
| 探索层 | 动态参数策略 | 后期 contextual bandit/RL |

### 4.3 为什么不首先使用端到端神经接收机

端到端“原始 IQ → 文件比特”模型存在以下风险：

1. 需要覆盖极大量的 SNR、干信比、频偏、采样偏差、多径、增益和未知对手波形；
2. 仿真到实机存在明显域差异；
3. 模型失败时难以定位是同步、相位、均衡还是译码问题；
4. 数据量、训练时间和硬件资源要求高；
5. 实时推理和缓存更难保证；
6. 对训练分布外的干扰可能突然完全失效；
7. 很难获得比成熟同步、匹配滤波和 FEC 更可靠的保底性能。

因此，本项目把机器学习放在“识别环境与选择工具”位置，而不是让它重新发明整个接收机。

## 5. 传统通信基线设计

### 5.1 实现顺序

严格按以下顺序推进：

1. 无噪声 BPSK 基带回环；
2. AWGN 下 BPSK，验证理论 BER 趋势；
3. 加入固定帧结构、CRC 和文件分块；
4. 加入根升余弦成形与匹配滤波；
5. 加入定时偏差和符号同步；
6. 加入载波频偏、相位偏差和载波同步；
7. 加入 FEC 与交织；
8. 升级到 QPSK；
9. 加入多径和均衡；
10. 加入各类干扰；
11. 接入记录的真实 IQ；
12. 接入 SDR 实时收发；
13. 最后才接入机器学习决策。

每一步都必须保留上一阶段的自动回归测试。

### 5.2 推荐的第一版波形

第一版稳健基线建议采用单载波 BPSK，稳定后升级为单载波 QPSK：

- 优点：结构直观、同步链条明确、峰均比较低、易于调试；
- 缺点：面对频率选择性衰落或多个窄带干扰时，灵活性可能不如 OFDM；
- 定位：它是必须存在的可靠保底方案，不代表最终一定最优。

OFDM 作为第二条候选波形，仅在以下条件满足后开发：

- 单载波基线已经通过完整测试；
- 实测显示频率选择性衰落或窄带干扰是主要瓶颈；
- CPU 能承担 FFT、信道估计和均衡；
- PAPR、同步和频偏问题已被纳入测试；
- 规则允许所需占用带宽和子载波结构。

### 5.3 建议帧结构

以下结构仅为可配置起点，具体长度必须通过测试决定：

```text
| Guard | AGC/Wake-up | Preamble | Sync Word | Robust Header |
| Pilot/Channel Estimation | Coded Payload | CRC | Guard |
```

#### 字段职责

- `Guard`：减轻相邻突发和滤波瞬态影响；
- `AGC/Wake-up`：让接收增益和检测器稳定；
- `Preamble`：粗频偏、定时、相关检测；
- `Sync Word`：确认帧边界并降低误检；
- `Robust Header`：使用固定的强健调制与编码，携带协议版本和有效载荷参数；
- `Pilot`：信道、相位或均衡器估计；
- `Coded Payload`：经过 FEC 和交织的数据；
- `CRC`：判断该块能否计入正确数据；
- 末尾 `Guard`：避免连续突发间互相污染。

#### 建议包头字段

| 字段 | 说明 |
|---|---|
| Protocol version | 防止不同版本误解码 |
| File/session ID | 区分本次随机文件 |
| Block sequence | 数据块序号 |
| Total blocks 或 file size | 文件重组 |
| Payload length | 处理尾块 |
| PHY profile ID | 指明调制、编码、交织参数 |
| Repetition ID | 识别重复副本 |
| Header CRC | 防止错误包头污染重组 |

包头必须比有效载荷更鲁棒；否则接收端即使解出数据，也无法正确归档。

### 5.4 文件层策略

1. 将任意二进制文件按固定或可配置块长切分；
2. 每块添加文件 ID、序号、有效长度和 CRC；
3. 接收端只接受通过包头 CRC 和载荷 CRC 的块；
4. 使用 `(file_id, block_seq)` 去重；
5. 允许乱序接收；
6. 维护位图记录已恢复块；
7. 保存阶段性恢复结果，进程异常后能够继续；
8. 对未收齐的文件，依据评分规则决定提交完整前缀、正确块集合或重构文件；
9. 不依赖随机文件可压缩；
10. 如果允许喷泉码或无反馈冗余，可在稳定分块协议后进行独立对照试验。

### 5.5 同步链路

接收端至少需要区分并记录以下状态：

```text
IDLE → ENERGY_DETECTED → PREAMBLE_FOUND → CFO_CORRECTED
→ TIMING_LOCKED → HEADER_DECODED → PAYLOAD_DECODING
→ CRC_PASS/CRC_FAIL → SEARCH/RELOCK
```

推荐拆分：

- 能量或相关检测；
- 粗载波频偏估计；
- 粗频偏校正；
- 符号定时恢复；
- 精细载波/相位跟踪；
- 帧同步；
- 导频信道估计；
- 均衡；
- 软判决解调；
- FEC 译码；
- CRC 校验。

不得把所有同步步骤写成一个难以测试的巨型函数。

### 5.6 纠错与交织路线

建议分阶段比较：

1. 无编码基线；
2. 重复码仅用于调试，不作为长期方案；
3. 卷积码 + Viterbi：实现和调试相对直接；
4. LDPC：在基线稳定后评估吞吐率、译码迭代和 CPU 成本；
5. Polar 或喷泉码：仅在规则与时间允许时研究。

交织器重点应对突发干扰，但会增加时延和缓存。需要比较：

- 不交织；
- 短块交织；
- 深交织；
- 不同突发占空比和持续时间下的 Goodput。

### 5.7 候选物理层配置档位

档位仅作为实验模板，参数必须由规则和测试补全：

| Profile | 用途 | 候选调制 | 编码强度 | 目标 |
|---|---|---|---|---|
| P0 Rescue | 极差环境保底 | BPSK | 强 | 降低完全失联概率 |
| P1 Robust | 常规稳健 | BPSK/QPSK | 中强 | 提高低分位 Goodput |
| P2 Balanced | 默认档 | QPSK | 中 | 平衡吞吐率和可靠性 |
| P3 Fast | 良好信道 | QPSK/更高阶 | 弱 | 提高峰值吞吐率 |
| P4 Notch | 窄带干扰 | 与基线相同 | 可配置 | 使用陷波或频域屏蔽 |
| P5 Burst | 突发干扰 | BPSK/QPSK | 中强 + 深交织 | 对抗成片错误 |

若无反馈能力，发射端不得依据接收端不可知状态随意切换档位。可以使用固定档位、预定义时间表或鲁棒包头指示模式，但必须通过专门试验证明不会导致更多失锁。

## 6. 信道与干扰仿真器

### 6.1 设计目标

仿真器必须做到：

- 每种损伤可以独立开关；
- 参数、单位和随机种子显式记录；
- 同一输入和种子产生一致输出；
- 可以输出中间 IQ、图表和真值标签；
- 可以单独验证每个信道模块；
- 能够组合多个损伤，但组合顺序明确；
- 支持批量 Monte Carlo 测试；
- 支持从仿真切换到实测 IQ 而不更改接收机上层接口。

### 6.2 必须实现的损伤

#### 基础信道

- AWGN；
- 固定增益与随机幅度衰落；
- 静态相位偏移；
- 载波频率偏移 CFO；
- 采样时钟偏差 SRO；
- 分数符号定时偏差；
- 相位噪声；
- 多径 FIR 信道；
- 缓慢时变增益或衰落；
- ADC 量化；
- 削顶和非线性；
- DC 偏置和 IQ 不平衡；
- 丢样、重复样本或缓冲区不连续，用于模拟工程故障。

#### 干扰类型

- 窄带单音；
- 多音干扰；
- 扫频/chirp；
- 宽带高斯噪声；
- 带限噪声；
- 突发噪声；
- 邻频数字信号；
- 同频 BPSK/QPSK/QAM；
- OFDM 类干扰；
- 周期性或随机占空比干扰；
- 多干扰叠加。

不得假设现场干扰一定属于训练类别。必须保留 `unknown/out-of-distribution` 路径。

### 6.3 建议的初始测试参数网格

以下仅为离线压力测试起点，不代表赛场参数：

- SNR：`[-12, -9, -6, -3, 0, 3, 6, 9, 12, 18, 24] dB`；
- INR：`[-20, -10, 0, 10, 20, 30] dB`；
- CFO/符号率：`[0, 1e-4, 1e-3, 5e-3, 1e-2, 5e-2]`；
- 分数定时偏差：`[0, 0.1, 0.25, 0.5, 0.75, 0.9] symbol`；
- 突发占空比：`[0.05, 0.1, 0.25, 0.5, 0.8]`；
- 突发长度：按 `[1, 4, 16, 64, 256] symbols` 起步；
- 多径：1、2、4、8 taps，功率延迟谱可配置；
- 干扰频偏：覆盖带内中心、带内边缘、邻频和带外；
- 信号幅度：覆盖 AGC 正常区、低电平区和削顶区。

开发阶段不应直接运行完整笛卡尔积。采用三层测试：

1. `smoke`：少量典型点，快速运行；
2. `regression`：覆盖主要边界，每次合并前运行；
3. `campaign`：大规模 Monte Carlo，里程碑或夜间运行。

### 6.4 信道模块接口建议

```python
class ChannelModel(Protocol):
    def apply(
        self,
        iq: np.ndarray,
        sample_rate_hz: float,
        rng: np.random.Generator,
        config: "ChannelConfig",
    ) -> "ChannelOutput": ...
```

`ChannelOutput` 至少包含：

- `iq: complex64 array`；
- `ground_truth`：实际 SNR、INR、CFO、定时偏差、信道冲激响应；
- `events`：突发起止、削顶、丢样等；
- `seed`；
- `config_hash`。

## 7. 机器学习模块

### 7.1 机器学习任务优先级

| 优先级 | 任务 | 推荐模型 | 风险 | 是否进入首版 |
|---:|---|---|---|---|
| 1 | 干扰类型分类 | 随机森林 | 低 | 是 |
| 2 | 干扰存在与否/未知检测 | 阈值 + 一类检测 | 中 | 是 |
| 3 | 接收配置选择 | 规则表/监督学习 | 中 | 是 |
| 4 | 信道质量/Goodput 预测 | 梯度提升/MLP | 中 | 可选 |
| 5 | 原始 IQ 一维 CNN 分类 | 小型 1D CNN | 中 | 第二版 |
| 6 | 时频图 CNN 分类 | 小型 2D CNN | 中高 | 可选 |
| 7 | 神经均衡器 | MLP/1D CNN | 高 | 后期研究 |
| 8 | 强化学习动态策略 | Bandit/PPO/DQN | 高 | 不阻塞主线 |
| 9 | 端到端神经解调 | 深度网络 | 极高 | 默认不做 |

### 7.2 第一版特征集合

对每个固定时间窗或固定符号窗提取：

#### 时域特征

- 平均功率、方差、RMS；
- 峰值、峰均比 PAPR；
- 实部/虚部均值和方差；
- 幅度均值、方差、偏度、峰度；
- 相位差统计；
- 过零率或符号变化率；
- 包络自相关峰；
- 削顶比例；
- 突发占空比估计。

#### 频域特征

- Welch PSD；
- 总带内功率；
- 分段子带功率；
- 最大谱峰高度和位置；
- 前若干谱峰间距；
- 频谱质心、带宽、滚降点；
- 频谱平坦度；
- 频谱熵；
- 带边功率比；
- 邻带/主带功率比；
- 窄带峰数量；
- 扫频轨迹的粗略变化率。

#### 通信接收机特征

- 前导相关峰值及峰旁比；
- 粗 CFO 估计；
- 定时误差方差；
- 相位环误差；
- EVM；
- 软判决 LLR 绝对值均值和分位数；
- FEC 译码失败或迭代次数；
- 最近滑动窗口 PER；
- 最近 Goodput；
- AGC 增益与削顶状态；
- 重同步次数。

所有特征必须带有：

- 数学定义；
- 单位；
- 计算窗口；
- 缺失值策略；
- 归一化方法；
- 对采样率和窗长的依赖说明。

### 7.3 标签体系

推荐分层标签，而不是只有一个平面类别：

```text
interference_present: true/false
interference_family:
  clean | tone | multitone | sweep | broadband_noise |
  bandlimited_noise | burst | digital_single_carrier |
  ofdm_like | mixed | unknown
spectral_relation:
  inband | band_edge | adjacent | out_of_band
temporal_pattern:
  continuous | bursty | periodic | unknown
severity:
  low | medium | high | extreme
```

严重程度应尽可能依据对基线 Goodput 的影响定义，而不仅依据绝对功率。

### 7.4 数据集单元与格式

每个样本至少包含：

```text
sample_id
capture_id
scenario_id
split_group_id
source: simulated/recorded
seed
timestamp
band
center_frequency_hz
sample_rate_hz
rf_bandwidth_hz
window_start_sample
window_length_samples
desired_waveform_profile
snr_db
inr_db
cfo_hz
sro_ppm
timing_offset_symbols
channel_taps
interference_labels
feature_vector_version
raw_iq_reference
code_version
config_hash
```

建议存储：

- 小型数据：压缩 `.npz` 分片 + `.jsonl/.csv` 元数据；
- 大型连续 IQ：原始 `complex64` 文件 + 独立元数据；
- 模型输入特征：Parquet/CSV；
- 禁止把大型原始 IQ 提交到 Git；
- 每个数据集版本生成 manifest 和校验和。

### 7.5 数据划分与泄漏防护

这是机器学习部分最容易出现的严重错误。

必须按完整的 `capture_id/scenario_id` 分组划分训练、验证和测试集。来自同一次连续采集的相邻 IQ 窗口高度相关，不能被随机打散到不同集合。

至少设置：

- `train`：模型拟合；
- `validation`：阈值、超参数和策略选择；
- `test_seen_domain`：已知类型但未见参数组合；
- `test_unseen_parameter`：训练范围外的 SNR/INR/CFO；
- `test_unknown_interference`：训练中未出现的干扰；
- `test_real_capture`：真实 SDR 数据；
- `test_end_to_end`：比较最终 Goodput。

不得在测试集上反复调参后继续把该集合称为“测试集”。一旦用于选择模型，应升级为开发集并创建新的盲测集合。

### 7.6 第一版模型：随机森林

推荐原因：

- 对不同尺度特征相对不敏感；
- 能处理非线性边界；
- 小数据条件下较稳；
- 推理速度快；
- 可输出特征重要性；
- CPU 部署简单；
- 比深度模型更容易定位错误。

必须与以下基线比较：

1. 恒定预测多数类；
2. 简单功率阈值；
3. 人工规则分类器；
4. 逻辑回归；
5. 随机森林。

模型指标至少包含：

- 混淆矩阵；
- macro F1；
- 每类 precision/recall；
- 置信度校准；
- 未知干扰下的错误置信度；
- 单窗推理延迟；
- 端到端策略 Goodput 提升。

### 7.7 第二版模型：小型 1D CNN

输入形状建议：

```text
[batch, channel=2, samples]
channel 0 = real(I)
channel 1 = imag(Q)
```

设计约束：

- 参数量尽量控制在百万级以下，首选更小；
- 支持 CPU 推理；
- 使用全局池化减少参数；
- 训练中随机化幅度、相位和时间平移；
- 不让模型依赖固定绝对幅度或特定采集位置；
- 保存归一化方式和输入窗长；
- 与随机森林比较 Goodput，而不是只比较分类准确率。

只有当 1D CNN 在真实盲测和端到端 Goodput 上稳定超过随机森林，并满足实时预算时，才允许成为默认模型。

### 7.8 置信度门控与回退

机器学习输出不得直接无条件控制接收机。

建议流程：

```python
prediction = classifier.predict(features)

if model_missing_or_failed:
    profile = ROBUST_DEFAULT
elif prediction.confidence < confidence_threshold:
    profile = ROBUST_DEFAULT
elif is_out_of_distribution(features):
    profile = ROBUST_DEFAULT
else:
    profile = policy.lookup(prediction, receiver_metrics)

profile = hysteresis_and_rate_limit(profile)
```

必须具备：

- 最低置信度阈值；
- OOD/未知检测；
- 最小保持时间；
- 切换滞回；
- 单位时间最大切换次数；
- 参数合法范围裁剪；
- 模型超时回退；
- 策略应用失败回退；
- 回退事件日志。

## 8. 自适应策略控制器

### 8.1 不依赖反馈的接收端动作

在仅单向传输的情况下，优先允许机器学习调整：

- 数字带通滤波器类型和带宽；
- 陷波器开关、中心频率和带宽；
- AGC 模式、目标电平和冻结时机；
- 帧检测阈值；
- 同步环带宽；
- CFO 搜索范围和步长；
- 均衡器类型、阶数、步长和训练长度；
- 软判决缩放；
- FEC 最大迭代次数；
- 重同步策略；
- 是否暂缓解码并积累更多样本；
- 多候选解码器的执行顺序。

这些动作仍需确认是否符合规则及实时预算。

### 8.2 依赖反馈或收发协同的动作

只有规则明确允许反馈时，才考虑：

- 调制阶数；
- FEC 类型和码率；
- 交织深度；
- 包长；
- 重复次数；
- ARQ/HARQ；
- 发射频点；
- 发射带宽；
- 发射增益/功率；
- 导频密度；
- 波形在单载波与 OFDM 间切换。

若允许反馈，反馈控制包必须：

- 使用最鲁棒的固定波形；
- 带序号、CRC 和超时；
- 丢失时保持上一个安全配置；
- 防止旧控制包回滚配置；
- 防止来回振荡；
- 统计反馈占用的时间和频谱开销。

### 8.3 策略映射初稿

| 环境判断 | 接收端建议 | 发射端建议（仅反馈允许） | 风险 |
|---|---|---|---|
| Clean/高 SNR | 默认滤波、低迭代 | 提高吞吐档位 | 过度激进导致突发失败 |
| 窄带单音 | 自适应陷波 | 避开频点或用频域屏蔽 | 陷波损伤有用信号 |
| 邻频强干扰 | 缩窄滤波、检查 AGC | 调整中心频率 | 滤波过窄引入 ISI |
| 宽带噪声 | 鲁棒同步、更多译码迭代 | BPSK/强 FEC | 吞吐率下降 |
| 突发干扰 | 交织对应解码、短窗重同步 | 深交织/短包/重复 | 时延和冗余增加 |
| 多径 | 均衡器和导频跟踪 | 增加导频/CP | 估计误差和开销 |
| 未知/OOD | 回到稳健默认档 | 不切换或进入救援档 | 保守但可靠 |

每条策略必须通过 A/B 对照验证：

```text
相同信号、相同干扰、相同随机种子
固定默认策略 vs 自适应策略
比较 Goodput、失败率、CPU、切换次数
```

## 9. 强化学习与在线学习：后期可选路线

### 9.1 使用条件

只有同时满足以下条件才启动强化学习阶段：

- 传统基线已稳定；
- 监督式干扰分类与策略表已完成；
- 有可靠仿真环境；
- 动作空间由规则明确限定；
- 有可量化奖励；
- 有稳定回退策略；
- 剩余备赛时间充足；
- 证明规则表或监督学习存在明显性能瓶颈。

### 9.2 推荐先用 Contextual Bandit

如果每次选择参数后的影响主要体现在当前或短期吞吐率，而长期状态依赖较弱，优先使用 contextual bandit，而不是深度强化学习。

状态可以包括：

\[
s_t=[\widehat{SNR},\widehat{INR},EVM,PER,G_{t-k:t},\text{interference class},\text{confidence}]
\]

动作是有限个预先验证的安全 Profile，奖励为：

\[
r_t=\text{valid payload bits}-\lambda_1\text{switch cost}
-\lambda_2\text{failure penalty}
\]

### 9.3 深度强化学习建模

若状态转移和长期策略确实重要，可考虑：

- 离散动作：DQN/PPO；
- 连续参数：PPO/SAC/TD3；
- 多队动态环境：训练时做域随机化和对手随机化。

正式比赛策略必须：

- 关闭或极度限制随机探索；
- 只允许选择经过验证的安全 Profile；
- 设置最短保持时间和切换成本；
- 异常时立即回退；
- 不允许越过规则规定的频率、带宽和功率边界。

### 9.4 为什么不能在赛场自由探索

- 每次错误动作会直接损失有效数据；
- 其他队伍使环境非平稳；
- 比赛时长有限，样本效率不足；
- 在线学习可能把暂时干扰误认为长期规律；
- 策略可能频繁切换，造成双方失配；
- 很难在现场验证新策略没有灾难性失败。

因此，强化学习的主要价值应体现在离线仿真和策略预训练，而不是比赛现场从零学习。

### 10.3 硬件抽象原则

所有上层算法只依赖统一接口：

```python
class SDRDevice(Protocol):
    def configure_rx(self, config: "RxHardwareConfig") -> None: ...
    def configure_tx(self, config: "TxHardwareConfig") -> None: ...
    def receive(self, num_samples: int) -> np.ndarray: ...
    def transmit(self, iq: np.ndarray) -> None: ...
    def health(self) -> "DeviceHealth": ...
```

至少实现：

- `SimulatedSDRDevice`：纯软件；
- `FileReplaySDRDevice`：重放记录 IQ；
- `PlutoCompatibleSDRDevice`：实机；
- 可选 `GNUradioBridgeDevice`。

不得在 DSP、ML 或协议代码中散布设备 URI、中心频率或驱动专用字段。

### 11.1 建议核心接口

```python
class FrameEncoder:
    def encode(self, payload: bytes, metadata: "FrameMetadata") -> np.ndarray: ...

class Receiver:
    def process(self, iq: np.ndarray) -> list["DecodeResult"]: ...

class FeatureExtractor:
    def transform(self, iq: np.ndarray, rx_state: "RxState") -> np.ndarray: ...

class InterferenceClassifier:
    def predict(self, features: np.ndarray) -> "Prediction": ...

class AdaptivePolicy:
    def select(self, prediction: "Prediction", metrics: "RxMetrics") -> "RxProfile": ...

class FileAssembler:
    def accept(self, result: "DecodeResult") -> "AssemblyStatus": ...
```

`DecodeResult` 至少包含：

- 是否检测到帧；
- 包头是否通过 CRC；
- 载荷是否通过 CRC；
- 文件 ID 与块序号；
- 原始/译码后错误统计（仿真中）；
- SNR/EVM/CFO 等估计；
- 使用的 Profile；
- 模型预测和置信度；
- 各阶段耗时；
- 失败原因枚举，而不是只有 `False`。

任何空口发射命令在关键 RF 字段为 `null` 时必须拒绝运行，并输出明确缺失项。纯软件仿真不受此限制，但应使用单独的 simulation 配置。

### 12.2 配置优先级

```text
代码默认安全值
< 版本控制中的通用 YAML
< 本地硬件配置
< 命令行显式覆盖
```

每次运行必须把解析后的最终配置保存到实验输出目录，不能只保存原始 YAML。

### 12.3 单位规则

- 变量名显式写单位：`sample_rate_hz`、`cfo_hz`、`snr_db`；
- 时间内部统一用秒，采样位置用整数 sample index；
- 角度统一注明 rad/deg；
- 功率增益区分 dB、dBm 和线性幅度；
- 不允许以无单位的 `rate` 同时代表码率、采样率或编码率；
- 配置加载时检查 Nyquist、带宽、采样率和滤波器边界是否合法。

## 13. 分阶段实施计划与验收标准

时间为建议工作量，不代表硬性日程。阶段依赖比日期更重要。

### 阶段 P0：规则冻结与工程初始化（0.5–2 天）

#### 任务

- [ ] 获取完整规则、设备资料和评分细则；
- [ ] 填写 `contest_rules.yaml`；
- [ ] 建立仓库、目录、虚拟环境和测试框架；
- [ ] 编写最小 README、STATUS、DECISIONS；
- [ ] 实现配置加载、校验、随机种子和结构化日志；
- [ ] 验证两台电脑都能离线安装同一环境；
- [ ] 建立依赖锁定与环境检查脚本。

#### 交付物

- 可安装项目骨架；
- 赛制问题清单与已确认答复；
- `python -m pytest` 可运行；
- 环境验证报告。

#### 验收门槛

- 新电脑按文档能够复现环境；
- 未配置 RF 规则时，硬件发射命令安全拒绝；
- 同一随机种子生成一致测试数据；
- 日志包含时间、版本、配置哈希和种子。

### 阶段 P1：无噪声与 AWGN BPSK 基线（2–4 天）

#### 任务

- [ ] 字节到比特、比特到字节可逆转换；
- [ ] BPSK 映射和硬判决；
- [ ] AWGN 模块；
- [ ] 理想符号级链路；
- [ ] BER 统计；
- [ ] 理论趋势对照；
- [ ] 固定随机种子回归测试。

#### 验收门槛

- 无噪声下随机文件逐字节完全一致；
- 高 SNR 下 BER 接近 0；
- BER 随 SNR 提高单调下降，偏差需要解释；
- 至少 20 个种子不出现形状、类型或边界错误。

### 阶段 P2：组帧、CRC、成形与同步（4–8 天）

#### 任务

- [ ] 文件分块和帧头；
- [ ] 包头/载荷 CRC；
- [ ] RRC 成形与匹配滤波；
- [ ] 过采样；
- [ ] 帧检测；
- [ ] CFO 估计与校正；
- [ ] 定时恢复；
- [ ] 相位跟踪；
- [ ] 文件去重和重组；
- [ ] 失败原因日志。

#### 验收门槛

- 无噪声但含随机时间偏移和合理 CFO 时完整恢复文件；
- 帧头损坏不会污染文件重组；
- 重复包只计一次；
- 任意二进制内容，包括全 0、全 1 和随机字节，都能正确处理；
- 误检、漏检和捕获时间有自动报告。

### 阶段 P3：FEC、交织和 QPSK（4–8 天）

#### 任务

- [ ] 至少一种 FEC；
- [ ] 软判决接口；
- [ ] 可配置交织器；
- [ ] QPSK 映射、解映射和相位歧义处理；
- [ ] 与无编码/BPSK 对照；
- [ ] Profile 配置系统。

#### 验收门槛

- 同等信道条件下给出编码增益与开销对照；
- 不能只报告 BER，必须报告 Goodput；
- 突发干扰下交织方案比不交织有可重复收益，或记录其不适用条件；
- QPSK 在中高 SNR 下提供更高 Goodput，低 SNR 下可回退。

### 阶段 P4：复杂信道、干扰库与压力测试（4–7 天）

#### 任务

- [ ] 实现第 6 节损伤和干扰模块；
- [ ] 构建 smoke/regression/campaign 三层测试；
- [ ] 输出瀑布图、PSD、星座图、相关峰和误码曲线；
- [ ] 建立基线性能地图；
- [ ] 找出主要失败模式和临界区。

#### 验收门槛

- 每种损伤有独立单元测试；
- 同一场景可精确重放；
- 批量测试中不存在未解释 NaN/Inf、数组越界或静默丢帧；
- 形成“场景 → 最优已知 Profile”的初始标签表。

### 阶段 P5：干扰数据集与随机森林（4–7 天）

#### 任务

- [ ] 实现特征提取；
- [ ] 生成仿真数据集；
- [ ] 实现分组划分和泄漏检查；
- [ ] 训练多数类、阈值、逻辑回归和随机森林基线；
- [ ] 进行置信度校准；
- [ ] 实现 OOD/低置信度回退；
- [ ] 评估推理延迟。

#### 验收门槛

- 数据 manifest 完整；
- 训练与测试不存在相同 capture/scenario 泄漏；
- 随机森林在盲测上优于简单基线；
- 单窗推理满足实时预算；
- 未知干扰下不会高置信度频繁选择危险策略；
- 模型文件包含特征版本、训练配置和校验和。

### 阶段 P6：自适应接收机 A/B 验证（3–6 天）

#### 任务

- [ ] 实现规则策略和 Profile 切换；
- [ ] 实现滞回、限频和回退；
- [ ] 将分类器接入但保留旁路；
- [ ] 完成固定策略 vs 自适应策略的成对实验；
- [ ] 分析收益来自哪里、在哪些场景有负收益；
- [ ] 调整为“只在有证据的场景切换”。

#### 验收门槛

- 同一测试集上报告平均、低分位和最坏 Goodput；
- 自适应方案不能显著提高完全失败率；
- 模型关闭或损坏时自动回到稳健默认档；
- 策略不出现高频抖动；
- 每次切换有原因、置信度和前后配置日志。

### 阶段 P7：SDR 驱动与实机闭环（5–10 天）

#### 任务

- [ ] 验证 NanoSDR 的实际接口和 Pluto 兼容程度；
- [ ] 接收连续 IQ 并保存；
- [ ] 发射已知测试波形；
- [ ] 在合法安全条件下完成有线/屏蔽回环；
- [ ] 测试采样率、缓存、丢样和时钟偏差；
- [ ] 完成真实文件传输；
- [ ] 采集真实数据并微调模型；
- [ ] 进行两台比赛电脑的离线部署演练；
- [ ] 处理 USB 断连和设备重连。

#### 验收门槛

- 可连续运行超过预计比赛时长的 2 倍，且无未处理崩溃；
- 无缓冲区持续溢出；
- 能保存并重放同一段真实 IQ；
- 有线/屏蔽条件下随机文件逐字节恢复；
- 实测性能差异有记录，模型在真实盲测上重新评估；
- 一台电脑或进程重启后有明确恢复流程。

### 阶段 P8：现场压力模拟与最终冻结（3–7 天）

#### 任务

- [ ] 多干扰源并发；
- [ ] 低信号、强邻频、突发和同频压力测试；
- [ ] 两台电脑全流程演练；
- [ ] 禁网环境演练；
- [ ] 冷启动、设备拔插和程序崩溃恢复；
- [ ] 自动结果目录和文件提交检查；
- [ ] 冻结依赖、模型、配置和操作手册；
- [ ] 准备稳健版和实验版两个启动入口。

#### 验收门槛

- 稳健版一条命令启动；
- 无网络环境全部功能可用；
- 设备异常有可理解提示，不会静默失败；
- 现场操作清单由非开发成员走通；
- 最终 bundle 有校验和和备用副本；
- 比赛前停止非必要升级。

### 阶段 P9：可选强化学习（不得阻塞 P8）

#### 任务

- [ ] 将 Profile 选择建模为 bandit/RL；
- [ ] 离线训练；
- [ ] 与规则策略成对比较；
- [ ] 实现严格动作掩码；
- [ ] 关闭正式赛场随机探索；
- [ ] 评估分布外干扰和非平稳性。

#### 采用门槛

只有在独立盲测上同时满足以下条件才可作为比赛候选：

- 平均 Goodput 有稳定提升；
- 第 5 百分位不恶化或可接受；
- 完全失败率不增加；
- 实时预算满足；
- 决策不违反规则；
- 回退可靠；
- 团队能解释、操作和排障。

## 14. 测试体系

### 14.1 测试金字塔

#### 单元测试

覆盖：

- 字节/比特转换；
- 调制映射；
- CRC 已知向量；
- 交织可逆；
- FEC 编解码；
- 滤波器响应；
- CFO 注入和估计；
- 信道模块功率标定；
- 特征数学定义；
- 配置校验；
- 文件去重和重组。

#### 集成测试

- TX → 理想信道 → RX；
- TX → AWGN → RX；
- TX → 组合损伤 → RX；
- 数据集 → 训练 → 推理；
- 分类器 → 策略 → RX 配置；
- SDR 文件重放 → RX；
- TX 文件 → 实机/回环 → RX 文件。

#### 回归测试

- 固定关键随机种子和预期范围；
- 对性能允许统计容差，不应写死单个浮点值；
- 保存曾导致崩溃或性能突降的最小场景；
- 每个修复必须增加对应回归用例。

#### 性能测试

- 每秒可处理样本数；
- 不同窗长的特征提取延迟；
- 模型推理延迟分布；
- FEC 迭代时间；
- 缓冲区积压；
- 长时间内存增长；
- 日志和绘图开销。

#### Hardware-in-the-loop

- 设备发现；
- 参数写入与读回；
- IQ 格式、幅度和符号顺序；
- 连续接收；
- 循环发送；
- USB 重连；
- 采样率与实际时钟偏差；
- 发射泄漏、接收削顶与 AGC；
- 两频段与对应滤波器/天线组合。

### 14.2 Monte Carlo 规则

- Smoke：每个关键场景至少 3 个种子；
- 日常回归：至少 20 个种子或达到稳定置信区间；
- 里程碑：关键场景建议 100 个以上种子；
- 极慢实机测试可减少次数，但必须报告样本数；
- 报告均值、标准差、中位数、5/95 百分位和完全失败率；
- 所有对比尽量使用相同随机种子成对测试。

### 14.3 正确性 Oracle

仿真中保留原始发送文件和每个数据块真值。接收结果以逐字节比较和 CRC 双重检查：

```python
correct_bytes = sum(
    1 for tx_byte, rx_byte in zip(tx_data, rx_data)
    if tx_byte == rx_byte
)
```

实际评分若按完整块计数，应使用与官方一致的 block-level oracle。不要用“看起来像文本”或文件能打开来判断正确性。

- PSD；
- spectrogram；
- 匹配滤波前后波形；
- 前导相关曲线；
- CFO 估计误差；
- 定时误差；
- 星座图；
- 均衡前后星座图；

- BER/PER/Goodput 曲线；
- 混淆矩阵；

## 16. 实时工程与稳定性要求

### 16.1 实时预算

若每个处理窗代表 \(T_w\) 秒信号，则平均处理时间必须显著小于 \(T_w\)：

\[
\rho=\frac{T_{\text{processing}}}{T_w}<1
\]

建议目标 \(\rho\le 0.5\sim0.7\)，为系统调度、日志、USB 抖动和短时峰值保留裕量。绝对阈值需依据比赛电脑实测确定。

### 16.2 数据流线程建议

至少逻辑分离：

1. SDR 采集线程；
2. 环形缓冲区；
3. DSP/解调工作线程；
4. 特征与 ML 低优先级线程；
5. 文件重组与磁盘写入；
6. 监控与日志。

不得让绘图、模型保存或大量日志阻塞采集线程。

### 16.3 缓冲区策略

- 有明确容量和延迟上限；
- 达到高水位时记录告警；
- 明确覆盖旧数据还是丢弃新数据；
- 每次丢样都记录位置和数量；
- 处理断点不得被当作连续相位数据；
- 支持离线重放相同断点场景。

### 16.4 健康监控

实时显示或记录：

- 设备连接状态；
- 最近一次收到样本的时间；
- 缓冲区深度；
- 丢样计数；
- 当前中心频率、带宽和增益；
- 同步状态；
- 最近 SNR/EVM/PER/Goodput；
- 当前 Profile；
- ML 分类、置信度和 OOD 状态；
- 恢复块数/总块数；
- CPU、内存和磁盘剩余空间；
- 最近异常。

监控界面不是首要交付，命令行文本状态也可以，但必须可诊断。

### 16.5 异常恢复

必须处理：

- 模型文件不存在或校验失败；
- SDR 未连接；
- 设备 URI 变化；
- USB 暂时断连；
- 接收返回空数组；
- IQ 出现 NaN/Inf；
- 采样率与配置不匹配；
- 缓冲区溢出；
- 文件写入失败；
- 磁盘空间不足；
- 包头异常长度；
- 模型推理异常；
- 策略给出非法参数；
- Ctrl+C 或程序意外退出。

接收端应周期性保存已通过 CRC 的数据块索引和内容，防止进程重启后全部丢失。

## 17. 仿真到实机的域迁移

### 17.1 主要域差异

- 实际噪声不一定是白噪声；
- SDR 存在 DC、IQ 不平衡、时钟误差和本振相噪；
- USB/驱动可能产生不连续样本；
- 实际 AGC 和增益链会改变幅度特征；
- 天线、滤波器和环境改变频响；
- 对手波形与模拟干扰不同；
- 现场多径与人员移动造成时变；
- 433 MHz 和 2.4 GHz 的传播、背景占用和硬件响应不同。

### 17.2 缩小域差异的方法

1. 仿真中做参数随机化；
2. 输入归一化，避免依赖绝对幅度；
3. 在训练中随机相位、时间平移和增益；
4. 注入 DC、IQ 不平衡、削顶和丢样；
5. 用真实 SDR 采集构建独立测试集；
6. 先评估，再决定是否加入真实数据微调；
7. 真实测试集按采集 session 分组；
8. 对两个频段分别报告性能；
9. 对未知干扰使用回退，而不是强行分类；
10. 比赛前保留原始真实 IQ，便于复盘。

### 17.3 实机数据采集矩阵

在合法、安全、可控条件下，逐步采集：

- 无线电静默背景；
- 仅期望信号；
- 不同接收增益；
- 不同信号电平；
- 不同 CFO/本振组合；
- 433 MHz + 对应滤波器/天线；
- 2.4 GHz + 对应滤波器/天线；
- 单音、邻频、突发和数字干扰；
- 不同距离、方向、极化和遮挡；
- 多人移动或环境变化；
- 长时间连续运行。

每次采集必须记录硬件、频率、带宽、增益、天线、滤波器、空间布置和时间。

## 21. 常见失败诊断树

### 21.1 完全收不到明显信号

1. 检查设备连接和 URI；
2. 检查 TX/RX 中心频率；
3. 检查采样率和模拟带宽；
4. 检查天线与滤波器频段是否匹配；
5. 检查发射是否真正启动；
6. 检查接收增益是否过低；
7. 检查 IQ 数据格式和缩放；
8. 使用频谱图而不是直接怀疑解调器；
9. 在安全有线/屏蔽条件下缩小问题范围。

### 21.2 有能量但检测不到帧

1. 检查前导码是否一致；
2. 检查频偏是否超出搜索范围；
3. 检查相关归一化；
4. 检查匹配滤波群时延；
5. 检查采样率和 samples-per-symbol；
6. 检查 AGC 是否在前导期间变化；
7. 画相关曲线和频谱；
8. 降为 BPSK Rescue 复测。

### 21.3 能检测帧但包头失败

1. 检查定时锁定；
2. 检查相位歧义；
3. 检查包头调制和编码 Profile；
4. 检查 bit/byte 顺序和端序；
5. 检查 CRC 定义；
6. 检查包头长度；
7. 对比发送和接收软比特。

### 21.4 包头通过但载荷 CRC 失败

1. 检查信道随时间变化；
2. 检查均衡和相位跟踪；
3. 检查交织/去交织；
4. 检查 FEC puncturing/码率；
5. 检查 payload 长度和尾块；
6. 检查突发干扰位置；
7. 检查 LLR 缩放；
8. 尝试更稳健 Profile 并做成对比较。

### 21.5 仿真很好、实机很差

1. 测量实际 CFO/SRO；
2. 检查削顶、DC 和 IQ 不平衡；
3. 检查驱动丢样；
4. 检查 AGC；
5. 检查模拟滤波器和天线；
6. 加入真实损伤到仿真；
7. 用真实 IQ 文件重放，分离硬件采集与算法问题；
8. 不要立即扩大深度模型，应先定位域差异。

### 21.6 ML 离线准确率高但系统无收益

1. 检查是否优化错目标；
2. 比较策略切换开销；
3. 检查分类正确后对应策略是否真的更优；
4. 检查数据泄漏；
5. 检查特征提取延迟；
6. 检查真实干扰是否属于训练类别；
7. 检查错误分类是否集中在高影响场景；
8. 用端到端 Goodput 重新训练策略或缩小使用范围。

## 27. 术语表

| 术语 | 含义 |
|---|---|
| SDR | 软件定义无线电，尽量用软件实现调制、滤波和协议 |
| IQ | 复基带信号的同相和正交分量 |
| Goodput | 单位时间内正确交付的有效载荷，不含协议冗余和错误数据 |
| BER | 比特错误率 |
| PER/FER | 包/帧错误率 |
| SNR | 信号功率与噪声功率之比 |
| INR | 干扰功率与噪声功率之比 |
| EVM | 误差矢量幅度，衡量星座点偏离程度 |
| CFO | 载波频率偏移 |
| SRO | 采样时钟偏差 |
| RRC | 根升余弦滤波器 |
| FEC | 前向纠错编码 |
| CRC | 循环冗余校验，用于检测错误 |
| AGC | 自动增益控制 |
| PAPR | 峰均功率比 |
| OOD | 分布外输入，训练中未覆盖的情况 |
| Profile | 一套经过验证的收发/接收参数组合 |
| HARQ | 混合自动重传请求，需要反馈支持 |
| HIL | Hardware-in-the-loop，硬件在环测试 |
| Monte Carlo | 多随机种子重复仿真以评估统计性能 |

## 28. 当前建议结论

1. **可以使用机器学习备战，而且值得做。**
2. 首要任务仍是建立可靠的传统数字通信基线。
3. 最优先的机器学习应用是干扰识别和接收策略选择。
4. 第一版使用手工特征 + 随机森林，不从大型神经网络开始。
5. 模型必须通过置信度门控、OOD 检测、滞回和稳健回退控制风险。
6. 所有算法最终都用正确数据 Goodput 评价。
7. 无反馈时重点优化接收机；允许反馈后才做动态调制、编码、换频和重传。
8. 强化学习适合离线仿真和后期研究，不应在正式比赛中从零探索。
9. 实机驱动、同步、缓存、丢样和长时间稳定性通常比模型结构更决定比赛结果。
10. 当前立即行动项是获取完整规则、设备资料和电脑配置，然后从纯软件 BPSK 文件回环开始。
