# SDR 通信对抗比赛：队员共享状态与操作预案

> 文档版本：2026-07-22  
> 适用对象：发射端操作员、接收端操作员、算法/协议开发、现场记录与安全负责人  
> 本文不包含个人用户名、盘符、本机安装目录或其他机器专属路径。

## 1. 项目目标

使用两台兼容 PlutoSDR 的 NanoSDR 开发板构建独立发射端和接收端，在比赛信道中传输主办方提供的随机二进制文件。

系统目标不是“频谱图好看”，而是在规定时间内最大化通过 CRC 验证、能够按序重组的正确有效字节数，并完整保留原始输入、原始 IQ、帧状态、缺块信息和最终输出。

## 2. 当前比赛合同基线

### 2.1 允许频率

主办方提供两种天线，因此团队合同只使用以下两个中心频率之一：

| 模式 | 中心频率 | 默认状态 | 必须使用的天线/滤波器 |
|---|---:|---|---|
| 433 MHz 模式 | `433000000 Hz` | **默认** | 433 MHz 天线及对应滤波器 |
| 2.4 GHz 模式 | `2400000000 Hz` | 备用 | 2.4 GHz 天线及对应滤波器 |

收发两端必须选择同一个模式。改变中心频率时，必须同时更换正确频段的天线和滤波器，不能只改软件变量。

### 2.2 其他规则

除上述二选一频率约定外，目前合同对以下内容不作额外限制：

- 调制方式、编码方式、帧结构、前导、导频和同步序列；
- CRC、FEC、交织、白化、重复发送和去重；
- BPSK、QPSK、扩频、OFDM 或其他自定义波形；
- 自适应参数选择、离线机器学习模型和本地日志；
- IQ 记录、频谱分析和赛后评分工具。

“合同不限制”不等于可以忽略射频安全。团队仍强制执行：有限发射、非循环默认、最低功率起步、正确天线/滤波器、禁止未经衰减的 TX-RX 直连，以及现场紧急停止机制。

### 2.3 团队内部默认参数

以下是当前基线，不是不可更改的最优值：

| 参数 | 默认值 | 说明 |
|---|---:|---|
| 中心频率 | `433000000 Hz` | 默认使用 433 MHz |
| 备用中心频率 | `2400000000 Hz` | 切换时两端和天线必须一起换 |
| 复基带采样率 | `2000000 sample/s` | TX/RX 必须一致 |
| RF 带宽 | `1000000 Hz` | 团队内部当前占用带宽上限 |
| 调制 | BPSK | 稳健基线，可切 QPSK |
| FEC | K=7、rate-1/2 卷积码 | 可关闭或替换，但两端必须一致 |
| 每符号采样数 | 8 | 与 RRC、同步器匹配 |
| RRC roll-off | 0.35 | 两端一致 |
| RRC span | 6 | 两端一致 |
| 前导长度 | 64 symbols | 协议 v1 固定值 |
| 保护间隔 | 16 symbols | 两端一致 |
| 文件块大小 | 256 bytes | 可调，需重新验证 |
| 确定性 seed | 42 | 两端一致 |
| RX 增益模式 | manual | 初始 30 dB，视现场调整 |
| TX 衰减 | 89.75 dB | 原生 Pluto 块的最低输出起点 |
| 单次测试时长 | 5 s | 软件始终保持有限时长 |
| Pluto buffer | 32768 samples | 观察 overflow/underflow 后再调 |

## 3. 技术栈

### 3.1 核心软件

| 组件 | 当前版本/用途 |
|---|---|
| RadioConda | 统一的离线 GNU Radio 运行环境 |
| GNU Radio | 3.10.12.0 |
| GNU Radio Companion | 编辑、生成和运行 `.grc` |
| `grcc` | 命令行编译 `.grc` |
| GNU Radio `gr-iio` | 原生 PlutoSDR Source/Sink |
| libiio | 0.26，设备发现、context 和底层诊断 |
| pyadi-iio | RadioConda 内可用，主要用于短脚本和辅助诊断 |
| Python | GRC 生成代码、协议和测试 |
| NumPy/SciPy | 波形、滤波、同步和 IQ 分析 |
| PyYAML | 赛规、安全门和配置读取 |
| pytest | 自动回归测试 |

### 3.2 使用原则

- 完整实时收发优先使用 GNU Radio + gr-iio；
- 设备发现和属性核验优先使用 libiio；
- pyadi-iio 只作为短脚本和辅助工具，不与运行中的 GNU Radio 同时占用同一设备；
- 比赛电脑使用冻结好的 RadioConda 环境，不在比赛现场安装或升级依赖；
- 不依赖网络、云服务或另一台电脑的侧信道才能完成协议闭环。

## 4. 系统架构

### 4.1 发射端

```text
默认占位复正弦 ─┐
                 ├─ 输入选择器 ─ 有限样本 Head ─ 双重授权检查 ─ PlutoSDR Sink
竞赛文件协议源 ─┘
```

竞赛文件协议链：

```text
文件
 -> 256 B 分块、文件 ID、序号、总块数和长度
 -> 帧头保护与 CRC16
 -> payload CRC32
 -> 可选卷积 FEC
 -> BPSK/QPSK
 -> RRC 脉冲成形
 -> 峰值缩放
 -> 非循环、有限 Pluto TX
```

默认占位输入为 100 kHz 复基带正弦，仅用于链路和频谱检查。正式比赛时切换到竞赛文件接口并指定输入文件。

### 4.2 接收端

```text
PlutoSDR Source
 -> 有限采集 Head
 -> 原始 IQ 文件
 -> 前导检测、同步、CFO/载波恢复
 -> 解调、FEC、CRC
 -> 去重、按序重组
 -> 输出文件和分类指标
```

CRC 未通过的块绝不进入最终文件。不完整时只保存已经验证的独立分块，不猜测或填补缺失字节。

### 4.3 离线组合工程

另外保留一份组合图，用于：

- 无硬件协议回归；
- AWGN、CFO、SRO、多径和干扰仿真；
- 固定 IQ 文件回放；
- BPSK/QPSK 和 FEC 参数对比。

真实 TX/RX 图不包含中间仿真信道。

## 6. 当前已知问题（必须如实汇报）

### 6.1 GRC 中仍保存了不可用的 IP URI

当前 TX/RX 工程默认使用网络 URI，但设备最近只通过 USB IIO 正常工作。使用旧 URI会出现：

```text
Unable to find host
Unable to create context
```

影响：流图在构造 Pluto Source/Sink 时退出，不会开始采样，也不会生成本轮输出。

处理：每次插入设备后先运行 `iio_info -S usb`，将 GRC URI 改为扫描返回值；只有一台设备时可用 `usb:`，两台可见时必须绑定精确 URI/序列号。

### 6.2 当前工程默认中心频率仍是 2.4 GHz

团队合同已改为默认 433 MHz，但现有 TX/RX GRC 尚未同步，当前保存值仍是 2.4 GHz。

影响：若直接运行，软件频率与团队默认策略不一致，并可能装错天线/滤波器。

处理：下一次工程修改必须同时把 TX、RX、生成器默认值和赛规配置改为 `433000000 Hz`，并重新执行全部测试。切换备用 2.4 GHz 时也必须两端同步。

### 6.3 TX 安全门目前必然关闭

三个 GRC 布尔开关默认关闭，赛规 YAML 的确认状态为 false，关键字段为空。因此运行 TX 会出现：

```text
RF TX gate closed
```

这是设计行为，不是程序错误。

处理：先完成频率、物理路径、有限时长和团队内部带宽约束，再填写配置。只有现场负责人确认后才打开三个布尔门；赛后立即恢复关闭。

### 6.4 尚未完成真实 RF 端到端验证

目前的 145 项测试证明结构、生成、协议和离线恢复正确，但不能替代：

- 实际 USB 持续流稳定性；
- TX underflow/RX overflow；
- 真实 CFO、SRO、DC、IQ 不平衡；
- 433 MHz/2.4 GHz 天线和滤波器效果；
- 两台独立设备之间的真实解码率。

在完成实机测试前，不得宣称比赛链路已经最终可用。

### 6.5 普通系统 Python 的 pyadi 环境不完整

普通 Python 虽安装 pyadi-iio，但缺少可见的原生 libiio 动态库，会在 `import adi` 时异常。RadioConda 内的 pyadi/libiio 可用。

处理：统一从 RadioConda 环境启动工具，不混用系统 pip、其他 Conda 和 RadioConda 二进制库。

### 6.6 USB URI 可能在拔插后变化

类似 `usb:1.3.5` 的拓扑地址不是永久常量。

处理：比赛开机、拔插、换 USB 口后重新扫描；记录序列号并绑定角色，不把枚举顺序当身份。

### 6.7 RX 输出只在成功采集并停止后完整写入

如果 Source 初始化失败，本次不会更新 IQ、metrics 或 decoded 文件。即使采集成功，没有完整合法帧时也不会生成最终解码文件。

处理：先检查文件更新时间，再看 raw IQ 和 metrics；不要仅以“没有 decoded 文件”判断程序崩溃。

### 6.8 同一设备可能被多个程序争用

GNU Radio、IIO Oscilloscope、pyadi 脚本或另一份流图同时打开同一 context，可能导致初始化失败、busy 或流异常。

处理：一个 SDR 同一时刻只交给一个主要程序；切换工具前先完整 Stop/Wait 并关闭旧进程。

### 6.9 原生 RX 块的带宽设置需要实机读回确认

当前 GNU Radio 版本的 Pluto Source 模板主要通过自动滤波参数配置接收链，GRC 中显示的带宽变量是否完全按预期写入，需要用 libiio 读回验证。

处理：启动 RX 后读取 `ad9361-phy` 的 `rf_bandwidth`、`sampling_frequency`、LO 和 gain mode，记录请求值与读回值。

### 6.10 当前固件不是标准干净版本字符串

设备报告固件 `v0.39-dirty`。目前 USB context 和 AD9363A 设备可识别，但必须确认比赛是否允许该固件，且不要在临近比赛时随意刷写。

处理：冻结当前可用固件；若必须统一版本，先备份、记录序列号和校准信息，在备用设备上验证后再操作。

## 7. 可能遇到的问题与预案

### 7.1 `Unable to create context`

按顺序处理：

1. 关闭所有可能占用 SDR 的软件；
2. 重新插拔数据 USB，并避免无供电/不稳定 Hub；
3. 运行 `iio_info -S usb`；
4. 若能看到设备，将 GRC URI 改为扫描值；
5. 执行 `iio_info -u usb:` 验证完整 context；
6. 若 Windows 能看到设备但 libiio 看不到，检查 USB 复合接口和驱动；
7. 不要把当前正常驱动随意交给 Zadig 替换；
8. 更换已知良好的数据线、USB 口和电脑进行交叉验证。

### 7.2 TX 报 `RF TX gate closed`

1. 确认当前是否真的要发射；仅软件测试不要解除；
2. 确认频率模式和对应天线/滤波器；
3. 确认有限时长和最低功率起步；
4. 填写团队赛规配置；
5. 打开 GRC 三个确认开关；
6. 重新 Generate；
7. 如果下一条错误是频率、带宽或时长不允许，修正配置而不是绕过检查。

### 7.3 RX IQ 全零、极弱或只有 DC

1. 检查 RX 天线端口和频段；
2. 核对 TX/RX 中心频率、采样率和带宽；
3. 对比 manual、slow_attack、fast_attack；
4. 检查 IQ 峰值、RMS、均值和饱和比例；
5. 使用已知占位单音，先证明频谱位置正确；
6. 再进入复杂文件解码，不要用解码器掩盖前端问题。

### 7.4 RX 严重饱和

1. 降低 RX manual gain；
2. 增加距离或有线衰减；
3. 增大 TX 衰减；
4. 检查是否装错滤波器或存在近场强干扰；
5. 记录削顶比例，禁止在削顶状态调协议参数。

### 7.5 检测到信号但无法解码

逐项核对两端：

```text
中心频率
采样率
调制
FEC
samples_per_symbol
RRC roll-off/span
前导长度
保护间隔
seed
```

然后查看分类指标：前导未检测、帧头失败、payload CRC 失败、重复帧、缺块。优先解决最前面的失败阶段。

### 7.6 TX underflow / RX overflow

1. 降低采样率；
2. 关闭不必要的 GUI sink 和实时绘图；
3. 增大 IIO buffer；
4. 原始 IQ 写盘和日志减少频率；
5. 使用主板直连 USB 口，关闭 USB 节能；
6. 记录首次错误发生时间，不只记录最终计数。

### 7.7 USB 中途断开

1. 立即停止流图，不反复自动重试 TX；
2. 检查供电、线缆和接口；
3. 重新扫描 URI；
4. RX 端保留已通过 CRC 的分块和现有 IQ；
5. 重新启动时不要覆盖前一轮证据，使用新的运行编号；
6. TX 重新发射前再次执行安全门确认。

### 7.8 433 MHz 和 2.4 GHz 切换

切换清单：

1. 停止 TX 和 RX；
2. 两端同时修改中心频率；
3. 更换两端对应天线和滤波器；
4. 确认 GRC、赛规配置和运行记录中的频率一致；
5. 先 RX-only 看环境频谱；
6. 最低 TX 输出做短时占位单音测试；
7. 确认频谱位置后再发送竞赛文件。

### 7.9 输出文件不完整

1. 查看 `missing_blocks` 和已验证分块目录；
2. 查看是否存在大量重复帧；
3. 调整重复策略、块大小或 FEC；
4. 不对缺块进行猜测填充；
5. 保存原始随机输入、接收输出和 SHA-256，按比赛评分规则提交。

### 7.10 比赛现场时间紧张

采用降级策略：

1. 首选已经验证的 433 MHz BPSK + 卷积码基线；
2. 禁止临场更换协议核心、固件或依赖环境；
3. QPSK、自适应和复杂抗干扰只有在基线稳定后启用；
4. 出现不明问题时回退到固定参数、短 buffer、manual gain 和占位单音；
5. 先恢复“可发现、可采集、频谱正确”，再恢复文件解码。

## 8. 现场标准操作流程

### 8.1 开赛前

1. 两台电脑离线冷启动；
2. 检查 RadioConda/GNU Radio 版本；
3. 分别扫描两台 SDR，记录 URI、序列号和角色；
4. 确认默认使用 433 MHz 天线和滤波器；
5. 两端核对完整参数表；
6. RX 先做有限采集，验证非全零、无饱和；
7. TX 使用最低输出和 5 秒以内占位单音；
8. 确认频谱位置后停止测试；
9. 记录比赛输入文件 SHA-256；
10. 将 TX 输入切换为竞赛文件。

### 8.2 开赛启动顺序

1. RX 端先启动；
2. 确认 RX 流图正在采样；
3. TX 负责人复核安全门和物理路径；
4. TX 启动有限发送；
5. 两端分别记录开始时间、URI、频率、增益和错误计数；
6. 不依赖另一台电脑的反馈才能保持程序运行。

### 8.3 停止与收尾

1. TX 自动结束后确认没有 cyclic buffer；
2. RX 到时停止并等待最终重组；
3. 保存原始 IQ、metrics、manifest、decoded 文件和 `.parts`；
4. 计算输入/输出 SHA-256；
5. 记录正确字节、缺块、重复帧、CRC 失败和 overflow/underflow；
6. 每一轮使用独立目录或运行编号，不覆盖上一轮数据。

### 8.4 紧急停止

出现错误频率、错误天线、过强功率、异常持续发射或设备过热时：

1. 立即停止 TX 流图；
2. 等待程序完成 stop/wait；
3. 如仍有载波，断开 TX 程序并断开设备供电；
4. 不在问题未定位前重新打开三个 TX 安全门；
5. 记录事件、参数和停止时间。

7. 完成 433 MHz 最低输出占位单音测试；
8. 完成有衰减/屏蔽的文件端到端测试；
9. 在不同增益、CFO、干扰和距离下记录门限；
10. 验证 2.4 GHz 备用链路；
11. 两电脑、两 SDR 做完整冷启动演练；
12. 冻结最终版本、配置和离线依赖。

## 11. 验收标准

在宣称“可以参赛”之前，至少满足：

- USB 冷启动后可稳定发现指定序列号；
- RX 能连续采集完整轮次，无持续 overflow；
- TX 使用有限、非循环模式，停止后载波消失；
- 433 MHz 模式两端参数与天线/滤波器一致；
- 占位单音频谱位置正确；
- 已知测试文件端到端逐字节一致；
- CRC 失败数据不会写入最终输出；
- 缺块、重复帧、CRC、overflow/underflow 均有记录；
- 紧急停止经过实际演练；
- 断网、禁用非必要接口后仍可完成全流程；
- 所有运行产物能按运行编号保存且不互相覆盖。

## 12. 队内沟通时必须保持一致的事实

- 当前软件和离线协议测试已通过，不等于真实 RF 全链路已经通过。
- USB IIO 最近可用，网络 IP IIO 不可用。
- 默认频率策略已经改为 433 MHz，但工程文件尚待同步。
- TX 安全门关闭是设计行为，不是 bug。
- 正式比赛优先使用最稳健、已实测的固定参数基线。
- 任何频段切换都必须同时修改 TX、RX、天线、滤波器和运行记录。
